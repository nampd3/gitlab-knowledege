# Requirements Document

## Introduction

The Go Analyzer Support feature extends the existing Project_Analyzer component of the Project Knowledge MCP Server so that Projects whose source code is written in Go yield Project_Profiles with the same depth and shape as Projects written in Python, JavaScript/TypeScript, or Java. The current Project_Analyzer is file-extension routed and produces near-empty Project_Profiles for Go repositories: the purpose summarizer does not consult `go.mod` or top-level package doc comments, the I/O extractor has no Go scanner so HTTP route handlers, scheduled tasks, message consumers and publishers, file I/O, and CLI entrypoints in Go code are all missed, the external service detector cannot identify the in-house Go client libraries used by this organization, and the database table detector recognizes none of the patterns that lead to SQL execution against the in-house Oracle access service.

This feature adds Go-aware behavior to each of the four sub-analyzers (purpose, I/O extractor, external services, database tables) and integrates the new behavior into the existing `Project_Analyzer.analyze()` aggregator, so a Go repository with no README and no language-agnostic signals (such as embedded URLs) produces a Project_Profile that names its purpose, lists its abstract inputs and outputs, names its external service dependencies, and lists its database table dependencies.

The acceptance criteria below are scoped to the concrete tech stack confirmed in four sample repositories on disk:

- `/root/fec_pool_service` — the in-house gRPC service that fronts Oracle. The `PoolAPI` gRPC service exposes a single `ExecuteQuery` RPC that accepts a `PoolExecuteQueryRequest` carrying a SQL `QueryString` field and returns the result as a JSON-encoded string. This repository is the canonical database-access mechanism for every other Go service in the organization.
- `/root/repayment_service` — an HTTP-per-request service. Bootstraps an `*http.ServeMux`, registers SOAP and pprof routes via `mux.Handle(...)` and `mux.HandleFunc(...)`, mounts a gRPC-gateway `runtime.ServeMux` at `/`, sends ActiveMQ messages via `*activemq.Sender.SendMessage(...)`, and reads from Oracle through the `fec_pool_service` adapter.
- `/root/cat-service` — a cron job service. Runs `cron.New(cron.WithSeconds())` from `github.com/robfig/cron/v3` (six-field schedule expressions with seconds precision), registers a single job with `c.AddFunc(<schedule>, <handler>)`, and reads from Oracle through the `fec_pool_service` adapter.
- `/root/aps_los_vtiger` — an ActiveMQ listener job service. Subscribes to ActiveMQ destinations through `*activemq.Receiver.Subscribe(ctx, handler, *domain.SubscriberConfig{Destination: ...}, retryOpts)`, dispatches to internal usecases, and reads from Oracle through the `fec_pool_service` adapter.

Every recognized call shape in the acceptance criteria below is grounded in code that appears verbatim in at least one of these four repositories. Where the underlying language or library would also support a call shape that does not currently appear in any of the four repositories (for example, the Go 1.22+ method-prefixed `http.ServeMux` pattern syntax, or direct `stomp.Dial(...)` calls instead of the in-house ActiveMQ wrapper), the Open Questions section at the end of this document records that the construction is supported by the language or library but is not currently observed in the user's stack.

The implementation language for the analyzer remains Python. The Go sub-analyzers SHALL NOT invoke a Go toolchain at runtime; Go source is parsed in-process by Python code.

## Glossary

- **Project_Analyzer**: The existing component (defined in the parent spec `project-knowledge-mcp`) that inspects a Project's contents and produces a Project_Profile. Reused unchanged in identity; this feature extends its behavior for Go.
- **Project_Profile**: The structured record produced by the Project_Analyzer for a Project, as defined in the parent spec.
- **Repository_Contents**: The in-memory snapshot of a Project's files at the analysis commit, as defined in the parent spec, that the Project_Analyzer consumes.
- **Source_Location**: A reference to a file path inside the Repository_Contents, optionally with a 1-indexed line number, that the Project_Analyzer attaches to detections.
- **Go_Source_File**: A file in the Repository_Contents whose path ends with the suffix `.go` (case-sensitive), excluding files under any directory segment named `vendor`.
- **Go_Module_Manifest**: A file in the Repository_Contents at the repository root whose name is exactly `go.mod`.
- **Go_Package_Doc_Comment**: The contiguous block of `//`-prefixed line comments or a single `/* ... */` block comment that immediately precedes the `package <name>` declaration in a Go_Source_File, with no blank line separating the comment block from the `package` line.
- **Go_HTTP_Framework**: The Go standard library `net/http` package, including the `*http.ServeMux` type and the package-level helpers `http.Handle`, `http.HandleFunc`, `http.NewServeMux`, and `http.ListenAndServe`. The four sample repositories do not use any third-party HTTP framework (Gin, Echo, Chi, Fiber, Gorilla mux, Fasthttp), so this glossary entry is intentionally narrow.
- **Go_Scheduler_Library**: Either the third-party library `github.com/robfig/cron/v3` (including major-version-suffixed paths) or the standard library `time.NewTicker` and `time.AfterFunc` constructions used at package or function scope.
- **Go_ActiveMQ_Client**: The in-house ActiveMQ wrapper at the import path `esb-go-libs/activemq`, together with its companion package `esb-go-libs/activemq/domain`. The wrapper exposes the types `*activemq.Client`, `*activemq.Sender`, and `*activemq.Receiver`, and the message and subscription configuration types `domain.Message` and `domain.SubscriberConfig`. The underlying transport is STOMP (`github.com/go-stomp/stomp/v3`), but the four sample repositories never reference the STOMP package directly; all ActiveMQ access goes through the wrapper.
- **Go_DB_Pool_Client**: The two import paths through which the four sample repositories reach the `fec_pool_service` gRPC service:
  - `fec_pool_service/pb` — the Protobuf-generated client package providing the `PoolAPIClient` interface, its `ExecuteQuery(ctx, *PoolExecuteQueryRequest, opts...)` method, and the `PoolExecuteQueryRequest` message type whose `QueryString` field carries the SQL statement.
  - `esb-go-libs/dbadapter` — the in-house adapter package providing the `IPoolAPIAdapter` interface, its `PoolExecuteQuery(ctx, *model.PoolServiceRequest, username, password)` method, and the request type `model.PoolServiceRequest` (defined in `esb-go-libs/dbadapter/model`) whose `QueryString` field carries the SQL statement. Three of the four sample repositories (`repayment_service`, `cat-service`, `aps_los_vtiger`) call the database exclusively through this adapter.
- **Go_File_IO_Function**: Any of the following calls into the Go standard library: `os.Open`, `os.Create`, `os.OpenFile`, `os.ReadFile`, `os.WriteFile`, `ioutil.ReadFile`, `ioutil.WriteFile`, `ioutil.ReadAll` invoked on the result of `os.Open`.
- **Go_DI_Framework**: The Uber `fx` dependency-injection framework at the import path `go.uber.org/fx`. All four sample repositories bootstrap their main binary with `fx.New(...)` and register components with `fx.Provide(...)`, `fx.Invoke(...)`, `fx.Module(...)`, and `fx.Hook{...}`. These calls describe internal wiring and are never CLI entry points, HTTP routes, message subscriptions, or external service dependencies.
- **Go_Config_Library**: The `github.com/spf13/viper` library. All four sample repositories construct a `*viper.Viper` instance with `viper.New()` and read configuration through `(*viper.Viper).GetString`, `(*viper.Viper).BindEnv`, `(*viper.Viper).SetConfigName`, `(*viper.Viper).AddConfigPath`, and the package-level equivalents on the global Viper instance. Configuration reads describe how the binary is parameterized at startup, not data the binary processes at runtime.
- **Go_Tracing_Library**: The Elastic APM Go agent at the import paths `go.elastic.co/apm/v2`, `go.elastic.co/apm/module/apmhttp/v2`, `go.elastic.co/apm/module/apmgrpc/v2`, and any other `go.elastic.co/apm/...` path. All four sample repositories register the APM tracer in `main` and wrap HTTP and gRPC handlers with APM middleware. The APM agent is observability infrastructure, not an external service the project depends on for behavior.
- **Go_CLI_Library**: The Go standard library `flag` package. The four sample repositories do not use `github.com/spf13/cobra` or `github.com/urfave/cli`, so this glossary entry is intentionally narrow.
- **Go_Toolchain**: Any external executable that performs Go compilation, formatting, linting, or static analysis, including but not limited to `go build`, `go run`, `go vet`, `gofmt`, `goimports`, and `staticcheck`.
- **Abstract_Input**, **Abstract_Output**, **External_Service_Dependency**, **Database_Table_Dependency**: As defined in the parent spec `project-knowledge-mcp`.

## Requirements

### Requirement 1: Recognize Go Source Files

**User Story:** As an MCP_Client, I want the Project_Analyzer to inspect Go source code, so that Projects written in Go produce a Project_Profile with detected inputs, outputs, dependencies, and tables instead of an empty Project_Profile.

#### Acceptance Criteria

1. WHEN the Project_Analyzer processes a Project whose Repository_Contents contains at least one Go_Source_File, THE Project_Analyzer SHALL include detections derived from those Go_Source_Files in the produced Project_Profile.
2. THE Project_Analyzer SHALL apply the Go sub-analyzers to a Go_Source_File regardless of the Go_Source_File's directory location in the Repository_Contents, including files under `cmd/`, `internal/`, `pkg/`, and any other directory.
3. THE Project_Analyzer SHALL NOT apply the Go sub-analyzers to any file whose path contains a directory segment named `vendor`, so that the vendored copies of third-party libraries observed in all four sample repositories do not contribute false detections.
4. WHEN the Project_Analyzer processes a Project whose Repository_Contents contains no Go_Source_File and no Go_Module_Manifest, THE Project_Analyzer SHALL produce the same Project_Profile as it would have produced before this feature was added.

### Requirement 2: Derive Purpose Summary from Go Sources

**User Story:** As an MCP_Client, I want the Project_Analyzer to derive a purpose summary from Go-specific sources when no README or repository description is available, so that Go Projects without a README still receive a non-"unknown" purpose summary when the source code carries enough metadata.

#### Acceptance Criteria

1. WHEN the Project_Analyzer processes a Project whose Repository_Contents contains a Go_Module_Manifest, THE Project_Analyzer SHALL treat the Go_Module_Manifest as one of the sources from which the purpose summary may be derived, in addition to the sources defined in the parent spec Requirement 3.2.
2. WHERE the Go_Module_Manifest contains a single-line comment immediately preceding or trailing the `module <module-path>` declaration whose comment body is non-empty after stripping the leading `//` and surrounding whitespace, THE Project_Analyzer SHALL select that comment body as the purpose-summary candidate contributed by the Go_Module_Manifest, in preference to the module-path fallback defined in Requirement 2.3.
3. WHERE no comment is associated with the `module` declaration in the Go_Module_Manifest, THE Project_Analyzer SHALL use the module path declared by the `module <module-path>` line, with any leading host-and-organization prefix removed (for example, `github.com/acme/payment-service` becomes `payment-service`; bare module names such as `repayment_service`, `cat-service`, `aps_los_vtiger`, and `fec_pool_service` observed in the four sample repositories are passed through unchanged), as the purpose summary candidate from the Go_Module_Manifest.
4. WHEN the Project_Analyzer processes a Project whose Repository_Contents contains a Go_Source_File at the repository root, at any path matching `cmd/main.go`, or at any path matching `cmd/<name>/main.go`, THE Project_Analyzer SHALL treat the Go_Package_Doc_Comment of that Go_Source_File as one of the sources from which the purpose summary may be derived, in addition to the sources defined in the parent spec Requirement 3.2.
5. THE Project_Analyzer SHALL apply the same length cap, whitespace normalization, and unknown-purpose fallback rules to a purpose summary derived from a Go source as the parent spec Requirements 3.3 and 3.4 define for the existing sources.
6. WHEN the Project_Analyzer can derive a purpose summary candidate from more than one source for the same Project, THE Project_Analyzer SHALL select among the candidates using a deterministic priority order whose relative position of the new Go sources is: README files, GitLab repository description, Go_Module_Manifest comment, Go_Module_Manifest module path, root-level package manifest description (`package.json`, `pyproject.toml`, `pom.xml`), top-level Python or JavaScript module docstring, Go_Package_Doc_Comment.
7. THE Project_Analyzer SHALL NOT use any string passed to a Go_Config_Library call (for example, the literal `<name>` argument of `viper.SetConfigName(<name>)` or the literal `<path>` argument of `viper.AddConfigPath(<path>)`) as a purpose summary candidate, because configuration file references describe deployment artefacts, not the project's purpose.

### Requirement 3: Extract HTTP Inputs and Outputs from Go Web Frameworks

**User Story:** As an MCP_Client, I want HTTP route registrations in Go code to appear as Abstract_Inputs and Abstract_Outputs on the Project_Profile, so that I can see what HTTP endpoints a Go Project serves.

#### Acceptance Criteria

1. WHEN the Project_Analyzer detects a route registration in a Go_Source_File for the Go_HTTP_Framework, THE Project_Analyzer SHALL record one Abstract_Input with category `http_request` describing the HTTP method (or "ANY" when the route registration accepts any method) and the route path, and one Abstract_Output with category `http_response` describing the HTTP method and the route path.
2. THE Project_Analyzer SHALL recognize as route registrations the following call shapes, all of which appear in the four sample repositories:
   - `http.HandleFunc(<pattern>, <handler>)` and `http.Handle(<pattern>, <handler>)` — package-level registrations on the default `http.DefaultServeMux`.
   - `<mux>.HandleFunc(<pattern>, <handler>)` and `<mux>.Handle(<pattern>, <handler>)` where `<mux>` is any identifier whose value is produced by `http.NewServeMux()`. Recognition of `<mux>` may be conservative: any identifier on which `.HandleFunc(...)` or `.Handle(...)` is called with a string-literal first argument inside a Go_Source_File that imports `net/http` SHALL be treated as a `*http.ServeMux` for the purposes of this requirement.
3. THE Project_Analyzer SHALL parse the route pattern argument according to the Go 1.22+ `http.ServeMux` pattern grammar:
   - WHEN the literal pattern matches the form `"<METHOD> <path>"` where `<METHOD>` is one of `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, or `OPTIONS` followed by a single space, THE Project_Analyzer SHALL split the literal into `<METHOD>` and `<path>` and record `<METHOD>` as the HTTP method and `<path>` as the route path.
   - WHEN the literal pattern does not match the method-prefixed form, THE Project_Analyzer SHALL record the HTTP method as "ANY" and the entire literal as the route path. (None of the four sample repositories currently uses the method-prefixed form; all observed registrations are bare paths such as `"/healthz"`, `"/readyz"`, `"/debug/pprof/"`, `"/soap/repayment-service"`, and `"/"`.)
4. WHEN the route path argument in a recognized route registration is not a string literal, THE Project_Analyzer SHALL still record an Abstract_Input and Abstract_Output for that registration, using a placeholder description naming the file path and the receiver identifier rather than a literal path.
5. THE Project_Analyzer SHALL acknowledge but SHALL NOT emit an Abstract_Input or Abstract_Output for HTTP server bootstrap calls of the form `http.ListenAndServe(<addr>, <handler>)`, `(*http.Server).ListenAndServe()`, or `(*http.Server).Shutdown(...)`. These calls govern the lifecycle of the HTTP server but do not declare new endpoints. THE Project_Analyzer SHALL silently suppress any Abstract_Input or Abstract_Output that would otherwise be emitted from a bootstrap call; THE Project_Analyzer SHALL NOT log, raise an error, or fail the analysis when it encounters one.
6. THE Project_Analyzer SHALL record on every Abstract_Input and Abstract_Output emitted by Requirement 3 a Source_Location identifying the Go_Source_File and the 1-indexed line number on which the route registration appeared.
7. WHEN the same route path and HTTP method combination is detected in more than one Go_Source_File or on more than one line of the same Go_Source_File, THE Project_Analyzer SHALL coalesce the detections following the same deduplication rule the I/O extractor already applies for non-Go languages, so that a single repository yields a focused Abstract_Input list and a focused Abstract_Output list.

### Requirement 4: Extract Scheduled Inputs from Go Schedulers

**User Story:** As an MCP_Client, I want scheduled task registrations in Go code to appear as Abstract_Inputs on the Project_Profile, so that I can see what scheduled events a Go Project consumes.

#### Acceptance Criteria

1. WHEN the Project_Analyzer detects a scheduled task registration in a Go_Source_File for any Go_Scheduler_Library, THE Project_Analyzer SHALL record one Abstract_Input with category `scheduled_event` describing the scheduler library and the schedule expression.
2. THE Project_Analyzer SHALL recognize as scheduled task registrations the following call shapes, the first three of which appear verbatim in the `cat-service` sample repository:
   - `cron.New(<options>...)` constructions, where `<options>` may include `cron.WithSeconds()`. WHERE the call to `cron.New(...)` includes the `cron.WithSeconds()` option, THE Project_Analyzer SHALL record on every Abstract_Input emitted for a scheduled task registered against the resulting `*cron.Cron` value an attribute indicating that schedule expressions are six-field (seconds-precision) cron expressions; otherwise THE Project_Analyzer SHALL record that schedule expressions are five-field (minute-precision) cron expressions.
   - `<cron-value>.AddFunc(<schedule>, <handler>)` and `<cron-value>.AddJob(<schedule>, <handler>)`, where `<cron-value>` is any identifier whose value is produced by a `cron.New(...)` call inside the same Go_Source_File.
   - `time.NewTicker(<duration>)` and `time.AfterFunc(<duration>, <handler>)` from the standard library `time` package, used at package or function scope.
3. WHEN the schedule expression in a recognized scheduled task registration is a string literal, THE Project_Analyzer SHALL include the literal schedule string verbatim in the Abstract_Input description without attempting to validate the expression's syntax (a six-field expression with seconds-precision SHALL be recorded exactly as it appears, even if a five-field cron parser would reject it). WHEN the schedule expression is not a string literal — for example, the `cat-service` repository's `cfg.JobCfg.CronSchedule` field-access expression — THE Project_Analyzer SHALL record the placeholder `<dynamic>` in the description.
4. THE Project_Analyzer SHALL record on every Abstract_Input emitted by Requirement 4 a Source_Location identifying the Go_Source_File and the 1-indexed line number of the recognized call.
5. WHEN the Project_Analyzer detects any call shape that resembles a Go_Scheduler_Library registration — for example, a method call on an identifier whose value is produced by `cron.New(...)` whose method name matches one of the patterns listed in Requirement 4.2 but whose schedule argument cannot be parsed, or a call that uses an unsupported scheduler whose method-name patterns match those listed in Requirement 4.2 — THE Project_Analyzer SHALL still record one Abstract_Input with category `scheduled_event` whose description records `<dynamic>` for the schedule and notes that the construction was malformed or unsupported. THE Project_Analyzer SHALL NOT silently skip a recognizable scheduler-call shape solely because its arguments are malformed or because the underlying scheduler library is not in the supported set.

### Requirement 5: Extract Message Consumer and Publisher Inputs and Outputs from the In-House ActiveMQ Wrapper

**User Story:** As an MCP_Client, I want ActiveMQ subscriber and publisher calls in Go code to appear as Abstract_Inputs and Abstract_Outputs on the Project_Profile, so that I can see which ActiveMQ destinations a Go Project consumes from and publishes to.

#### Acceptance Criteria

1. WHEN the Project_Analyzer detects a message consumer call in a Go_Source_File for the Go_ActiveMQ_Client, THE Project_Analyzer SHALL record one Abstract_Input with category `message_consumed` whose description names the library as `activemq` and includes the destination string when statically determinable, or `<dynamic>` otherwise.
2. WHEN the Project_Analyzer detects a message publisher call in a Go_Source_File for the Go_ActiveMQ_Client, THE Project_Analyzer SHALL record one Abstract_Output with category `message_published` whose description names the library as `activemq` and includes the destination string when statically determinable, or `<dynamic>` otherwise.
3. THE Project_Analyzer SHALL recognize as ActiveMQ consumer or publisher calls the following call shapes, each of which appears in the four sample repositories:
   - Consumer (observed in `aps_los_vtiger/internal/subscriber/subscriber.go`): a method call whose method name is `Subscribe` and whose third positional argument (after `ctx` and `handler`) is a composite literal of type `*domain.SubscriberConfig` or `&domain.SubscriberConfig{...}` from the package `esb-go-libs/activemq/domain`. The destination is the value of the `Destination` field of that composite literal.
   - Publisher (observed in `repayment_service/internal/usecase/repaymentservice_usecase.go`): a method call whose method name is `SendMessage` and whose fourth positional argument (after `ctx`, `transID`, and `correlationID`) is a composite literal of type `domain.Message` or `&domain.Message{...}` from the package `esb-go-libs/activemq/domain`. The destination is the value of the `Destination` field of that composite literal.
   - Connection setup (observed in `repayment_service/internal/adapter/adapter.go`): a call of the form `activemq.NewClient(&activemq.JmsConfig{BrokerUrl: <url>, ...})` from the package `esb-go-libs/activemq`. The Project_Analyzer SHALL acknowledge this call as the broker connection point but SHALL NOT itself emit an Abstract_Input or Abstract_Output from it; it MAY contribute to Requirement 8.5 (External_Service_Dependency for the ActiveMQ broker).
4. WHEN the value of the `Destination` field in a recognized `domain.Message` or `domain.SubscriberConfig` composite literal is a string literal, THE Project_Analyzer SHALL record the literal string verbatim in the description (for example, `REP.SERVICE.PAYMENT.ERR` from the `repayment_service` sample); THE Project_Analyzer SHALL NOT omit, redact, or silently drop a string-literal `Destination` value when one is present. WHEN the value is any other expression — including a field access such as `u.cfg.JMSConfig.PaymentErrQueue.Destination`, a function call, or a variable reference — THE Project_Analyzer SHALL record the placeholder `<dynamic>` in the description.
5. THE Project_Analyzer SHALL record on every Abstract_Input and Abstract_Output emitted by Requirement 5 a Source_Location identifying the Go_Source_File and the 1-indexed line number of the recognized call.

### Requirement 6: Extract File I/O Inputs and Outputs from Go Code

**User Story:** As an MCP_Client, I want file read and file write calls in Go code to appear as Abstract_Inputs and Abstract_Outputs on the Project_Profile, so that I can see which files a Go Project reads from disk and which files it writes.

#### Acceptance Criteria

1. WHEN the Project_Analyzer detects a Go_File_IO_Function call in a Go_Source_File whose use is a read (`os.Open`, `os.ReadFile`, `ioutil.ReadFile`, or `os.OpenFile` invoked with a flag bitmask whose first argument can be statically determined to contain `os.O_RDONLY` and not `os.O_WRONLY` or `os.O_RDWR`), THE Project_Analyzer SHALL record one Abstract_Input with category `file_read` describing the function called and the target path when statically determinable, or `<dynamic>` otherwise.
2. WHEN the Project_Analyzer detects a Go_File_IO_Function call in a Go_Source_File whose use is a write (`os.Create`, `os.WriteFile`, `ioutil.WriteFile`, or `os.OpenFile` invoked with a flag bitmask whose first argument can be statically determined to contain any of `os.O_WRONLY`, `os.O_RDWR`, `os.O_APPEND`, `os.O_CREATE`, or `os.O_TRUNC`), THE Project_Analyzer SHALL record one Abstract_Output with category `file_written` describing the function called and the target path when statically determinable, or `<dynamic>` otherwise.
3. WHERE the flag bitmask passed to `os.OpenFile` cannot be statically determined, THE Project_Analyzer SHALL record both an Abstract_Input with category `file_read` and an Abstract_Output with category `file_written` for that call.
4. THE Project_Analyzer SHALL record on every Abstract_Input and Abstract_Output emitted by Requirement 6 a Source_Location identifying the Go_Source_File and the 1-indexed line number of the recognized call.

### Requirement 7: Identify CLI Entry Points in Go Repositories

**User Story:** As an MCP_Client, I want the Project_Analyzer to identify the binary entry point of a Go Project so that I can see what executable the Project produces, even when no command-line argument parsing is present.

#### Acceptance Criteria

1. WHEN the Project_Analyzer detects a Go_Source_File at any path matching `cmd/<name>/main.go` (one directory deep under `cmd/`) containing a function declaration `func main()`, THE Project_Analyzer SHALL record one Abstract_Input with category `cli_argument` whose description names the binary as `<name>` derived from the directory segment.
2. WHEN the Project_Analyzer detects a Go_Source_File at the path `cmd/main.go` (the form used by all four sample repositories) containing a function declaration `func main()` and the Repository_Contents contains a Go_Module_Manifest, THE Project_Analyzer SHALL record one Abstract_Input with category `cli_argument` whose description names the binary using the last segment of the module path declared in the Go_Module_Manifest (for example, `repayment_service`, `cat-service`, `aps_los_vtiger`, or `fec_pool_service`).
3. WHEN the Project_Analyzer detects a Go_Source_File at the repository root containing a function declaration `func main()` and the Repository_Contents contains a Go_Module_Manifest, THE Project_Analyzer SHALL record one Abstract_Input with category `cli_argument` whose description names the binary using the last segment of the module path declared in the Go_Module_Manifest.
4. WHEN the Project_Analyzer detects in a Go_Source_File a call to a Go_CLI_Library function from the standard library `flag` package whose name is one of `flag.String`, `flag.StringVar`, `flag.Int`, `flag.IntVar`, `flag.Bool`, `flag.BoolVar`, `flag.Float64`, `flag.Float64Var`, `flag.Duration`, `flag.DurationVar`, `flag.Parse`, or `flag.NewFlagSet`, THE Project_Analyzer SHALL record one Abstract_Input with category `cli_argument` whose description names the framework as `flag` and includes the flag name when the call is one of the named registration functions and the flag-name argument is a string literal.
5. THE Project_Analyzer SHALL NOT emit an Abstract_Input from any Go_DI_Framework call, including `fx.New(...)`, `fx.Module(...)`, `fx.Provide(...)`, `fx.Invoke(...)`, `fx.Annotate(...)`, `fx.Hook{...}`, and any method call on `fx.Lifecycle`. These calls describe internal wiring of the binary, not its CLI surface.
6. THE Project_Analyzer SHALL record on every Abstract_Input emitted by Requirement 7 a Source_Location identifying the Go_Source_File and the 1-indexed line number of the recognized construction. IF the Project_Analyzer cannot determine a Source_Location (both file path and 1-indexed line number) for a recognized CLI construction, THEN THE Project_Analyzer SHALL NOT emit an Abstract_Input for that construction.

### Requirement 8: Detect External Service Dependencies in Go Code

**User Story:** As an MCP_Client, I want External_Service_Dependencies for Go Projects to include the in-house services the Go Project depends on, so that I can see when a Project reaches the `fec_pool_service` Oracle gateway, an ActiveMQ broker, or any service identified by a URL literal, regardless of whether a generic URL appears in the source.

#### Acceptance Criteria

1. THE Project_Analyzer SHALL emit Go-specific External_Service_Dependencies in addition to the existing language-agnostic URL-literal and SDK-pattern detections inherited from the parent spec Requirement 5. The Go-specific patterns are limited to those described in Requirements 8.2 through 8.5; the analyzer SHALL NOT attempt to detect AWS SDK, Google Cloud, Azure SDK, or Redis client patterns from Go source, because none of the four sample repositories use those libraries.
2. WHEN a Go_Source_File imports any package whose import path is `fec_pool_service/pb` (the Protobuf-generated client of the `PoolAPI` service) AND the file contains a method call whose method name is `ExecuteQuery` and whose receiver type can be resolved to `pb.PoolAPIClient`, THE Project_Analyzer SHALL emit one External_Service_Dependency with `kind = "other"` and `name = "fec_pool_service"`. WHEN the receiver type cannot be resolved with confidence, THE Project_Analyzer MAY treat the import alone as sufficient evidence to emit the dependency.
3. WHEN a Go_Source_File imports any package whose import path is `esb-go-libs/dbadapter` AND the file contains a method call whose method name is `PoolExecuteQuery`, THE Project_Analyzer SHALL emit one External_Service_Dependency with `kind = "other"` and `name = "fec_pool_service"`. The four sample repositories `repayment_service`, `cat-service`, and `aps_los_vtiger` each match this pattern through their `cfAppDbAdapter.PoolExecuteQuery(...)` calls.
4. THE Project_Analyzer SHALL NOT emit an External_Service_Dependency for any import whose path begins with `go.elastic.co/apm/`, including `go.elastic.co/apm/v2`, `go.elastic.co/apm/module/apmhttp/v2`, `go.elastic.co/apm/module/apmgrpc/v2`, and any other submodule. Elastic APM is a Go_Tracing_Library and represents observability instrumentation, not a service the project's behavior depends on.
5. WHEN a Go_Source_File contains a call of the form `activemq.NewClient(&activemq.JmsConfig{BrokerUrl: <url>, ...})` from the package `esb-go-libs/activemq`, THE Project_Analyzer SHALL emit one External_Service_Dependency with `kind = "message_broker"` and `name = "activemq"`. WHEN the value of the `BrokerUrl` field is a string literal whose host portion can be extracted, THE Project_Analyzer SHALL include that host in the dependency's `endpoint` attribute; otherwise THE Project_Analyzer SHALL record the endpoint as `<dynamic>`.
6. THE Project_Analyzer SHALL combine the External_Service_Dependencies emitted by Requirement 8 with the External_Service_Dependencies emitted by the existing language-agnostic detector following the parent spec Requirement 5.3 aggregation rule, so that the same external service detected from both a Go-specific pattern and a URL literal yields a single External_Service_Dependency entry whose `source_locations` list is the union of the detection sites.
7. THE Project_Analyzer SHALL record on every External_Service_Dependency emitted by Requirement 8 a Source_Location identifying the Go_Source_File and the 1-indexed line number of the recognized call or import.

### Requirement 9: Detect Database Table Dependencies in Go Code

**User Story:** As an MCP_Client, I want Database_Table_Dependencies for Go Projects to include tables identified by the SQL strings the Project sends to the `fec_pool_service` gRPC service, so that I can see which Oracle tables a Go Project reads or writes.

#### Acceptance Criteria

1. THE Project_Analyzer SHALL detect Database_Table_Dependencies in Go source by inspecting the SQL strings carried by `fec_pool_service` request constructions; the analyzer SHALL NOT attempt to detect GORM, ent, sqlc, or `database/sql` patterns from Go source, because none of the four sample repositories use those libraries.
2. WHEN a Go_Source_File contains a composite literal of type `model.PoolServiceRequest` from the import path `esb-go-libs/dbadapter/model` (the wrapper request type) and the literal sets the `QueryString` field, THE Project_Analyzer SHALL extract the value of the `QueryString` field as the SQL statement for analysis. WHEN the value is a string literal, raw string literal (backtick-quoted), or a call to `fmt.Sprintf(<format>, ...)` whose format argument is a string literal, the analyzer SHALL use that literal text (or the format text with `%v`, `%s`, `%d`, and similar verbs left in place) as the SQL statement.
3. WHEN a Go_Source_File contains a composite literal of type `pb.PoolExecuteQueryRequest` from the import path `fec_pool_service/pb` (the Protobuf-generated request type) and the literal sets the `QueryString` field, THE Project_Analyzer SHALL extract the value of the `QueryString` field as the SQL statement for analysis using the same rules as Requirement 9.2.
4. WHEN a Go_Source_File contains a composite literal of any in-house wrapper type that exposes a `QueryString` field and is passed to either `<adapter>.PoolExecuteQuery(...)` or `<executor>.Execute(...)` / `<executor>.ExecuteRaw(...)` (the pattern observed in `repayment_service/internal/repository/pool_executor.go` where `QueryOpts.QueryString` is forwarded into `model.PoolServiceRequest.QueryString`), THE Project_Analyzer SHALL extract the `QueryString` field value from the wrapper literal as the SQL statement for analysis using the same rules as Requirement 9.2.
5. WHEN the Project_Analyzer has extracted a SQL statement under Requirements 9.2, 9.3, or 9.4, THE Project_Analyzer SHALL parse the statement using the same regex-based table-name extraction the parent spec Requirement 6 already applies to other languages, attribute access mode according to the SQL keyword (`SELECT`, `JOIN` to `read`; `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `CREATE TABLE` to `write`), and emit one Database_Table_Dependency per distinct table name. WHERE the table name in the SQL is qualified by a schema prefix (for example, `APP_WS.IIB_PAYMENT_CONFIRMED` observed in `repayment_service`), THE Project_Analyzer SHALL record the table name in the form `<schema>.<table>` exactly as it appears in the SQL.
6. WHEN the Project_Analyzer has emitted any combination of `read` and `write` Database_Table_Dependency entries for the same table name — in any order, from any combination of Go-specific and language-agnostic detections — THE Project_Analyzer SHALL coalesce them into a single Database_Table_Dependency whose `access_mode` is `read_write`, following the parent spec Requirement 6.3 aggregation rule.
7. THE Project_Analyzer SHALL record on every Database_Table_Dependency emitted by Requirement 9 a Source_Location identifying the Go_Source_File and the 1-indexed line number of the composite literal carrying the `QueryString` field.
8. WHEN the Project_Analyzer has extracted a SQL statement under Requirements 9.2, 9.3, or 9.4 and a table name is identifiable but the SQL keyword cannot be matched to any of `SELECT`, `JOIN`, `INSERT`, `UPDATE`, `DELETE`, `MERGE`, or `CREATE TABLE`, THE Project_Analyzer SHALL still emit a Database_Table_Dependency for that table so the table reference is visible to downstream tools even when the access mode could not be determined. The parent spec's `DatabaseAccessMode` enum is currently closed at `read | write | read_write` (see `src/project_knowledge_mcp/models.py`), so the design phase SHALL resolve how to represent the unknown-access-mode case — for example, by adding an `unknown` value to the parent spec's enum, by selecting a conservative default such as `read_write`, or by suppressing this single dependency — without introducing an out-of-enum value at runtime. The chosen resolution SHALL be reflected in the parent spec before this requirement is implemented; see Open Questions item 6.

### Requirement 10: Pure-Python Implementation Without Go Toolchain

**User Story:** As an operator, I want the Project_Analyzer to analyze Go Projects without requiring a Go toolchain on the host, so that the MCP_Server can run in environments where no Go compiler is installed.

#### Acceptance Criteria

1. WHEN the Project_Analyzer processes a Go_Source_File or a Go_Module_Manifest, THE Project_Analyzer SHALL parse the file using only in-process Python code.
2. THE Project_Analyzer SHALL NOT invoke any Go_Toolchain during analysis of any Project.
3. WHEN the host on which the MCP_Server runs has no Go_Toolchain installed, THE Project_Analyzer SHALL produce the same Project_Profile for a Go Project that it would produce on a host that does have a Go_Toolchain installed.
4. WHEN the Project_Analyzer encounters a Go_Source_File whose interpretation requires toolchain-specific information that cannot be resolved by in-process Python parsing — specifically a build constraint (a `//go:build <expr>` line, or the legacy `// +build <expr>` line, that gates file inclusion) whose expression is anything other than empty or `!ignore`, or a `cgo` directive (a file containing `import "C"` together with the surrounding `// #cgo ...` comment lines and `/* ... */` C source blocks) — THE Project_Analyzer SHALL skip that Go_Source_File for the affected sub-analyzer and SHALL record the file path together with one of the reasons `"build constraint requires toolchain"` or `"cgo directive requires toolchain"` in the Project_Profile's `degraded_sections` field, using the existing aggregator failure-recording mechanism described in Requirement 11.2. THE Project_Analyzer SHALL NOT silently drop the file and SHALL NOT invoke any Go_Toolchain to resolve the constraint. The remaining Go_Source_Files in the Repository_Contents SHALL still be analyzed.

### Requirement 11: Resilience and Integration with the Existing Aggregator

**User Story:** As an operator, I want the Go sub-analyzers to fail gracefully and integrate cleanly with the existing aggregator, so that a malformed Go file does not poison the whole Project_Profile and so that adding Go support does not regress detection for Python, JavaScript/TypeScript, or Java Projects.

#### Acceptance Criteria

1. WHEN the Go-aware analysis raises any exception during the processing of a single Go_Source_File or the Go_Module_Manifest, THE Project_Analyzer SHALL contain the exception, skip that file, and continue analyzing the remaining files in the Repository_Contents.
2. WHEN the Go-aware analysis of a sub-analyzer (purpose, I/O extractor, external services, or database tables) raises an exception that is not contained by Requirement 11.1, THE Project_Analyzer SHALL record the corresponding section name in the Project_Profile's `degraded_sections` field following the existing aggregator behavior, populate the section with safe empty defaults, and continue producing the rest of the Project_Profile.
3. WHEN the Project_Analyzer processes a Project whose Repository_Contents contains both Go_Source_Files and source files in the languages already supported by the parent spec (Python, JavaScript/TypeScript, Java), THE Project_Analyzer SHALL emit detections from both the new Go sub-analyzers and the existing language-specific sub-analyzers, and SHALL coalesce duplicates following the parent spec Requirement 4 (I/O extractor deduplication), Requirement 5.3 (External_Service_Dependency aggregation), and Requirement 6.3 (Database_Table_Dependency aggregation) rules.
4. WHEN the Project_Analyzer is invoked twice on the same Repository_Contents, THE Project_Analyzer SHALL produce Project_Profiles whose `abstract_inputs`, `abstract_outputs`, `external_service_dependencies`, and `database_table_dependencies` lists are equal under list equality (same elements in the same order), so that the new Go sub-analyzers are deterministic.
5. WHEN the Project_Analyzer processes a Project whose Repository_Contents contains no Go_Source_File and no Go_Module_Manifest, THE Project_Analyzer SHALL produce a Project_Profile whose `abstract_inputs`, `abstract_outputs`, `external_service_dependencies`, and `database_table_dependencies` lists are equal under list equality to the lists the Project_Analyzer produced for the same Repository_Contents before this feature was added.

### Requirement 12: Treat uber/fx Dependency-Injection Calls as Internal Wiring

**User Story:** As an MCP_Client, I want the Project_Analyzer to ignore `uber/fx` dependency-injection registrations when extracting Abstract_Inputs, Abstract_Outputs, and External_Service_Dependencies, so that the Project_Profile reflects external behavior rather than the binary's internal wiring.

#### Acceptance Criteria

1. WHEN the Project_Analyzer encounters a call to any function or method exported by the Go_DI_Framework (`go.uber.org/fx`), including but not limited to `fx.New(...)`, `fx.Module(...)`, `fx.Provide(...)`, `fx.Invoke(...)`, `fx.Annotate(...)`, `fx.WithLogger(...)`, `fx.Hook{OnStart: ..., OnStop: ...}`, `(*fx.App).Run()`, `(fx.Lifecycle).Append(...)`, and the `fx.In` and `fx.Out` struct embeddings, THE Project_Analyzer SHALL treat the call as internal wiring and SHALL NOT emit any Abstract_Input, Abstract_Output, External_Service_Dependency, or Database_Table_Dependency from that call alone.
2. WHEN a function passed as an argument to `fx.Provide(...)` or `fx.Invoke(...)` itself contains constructions that match the patterns in Requirements 3 through 9 (for example, the `runScheduler` function passed to `fx.Invoke(runScheduler)` in `cat-service/cmd/main.go` calls `cron.New(cron.WithSeconds())` and `c.AddFunc(...)`), THE Project_Analyzer SHALL emit detections for those nested constructions according to the relevant requirement, with the Source_Location pointing to the nested construction rather than to the surrounding `fx.Invoke(...)` call.
3. THE Project_Analyzer SHALL recognize the `*fx.Lifecycle` type as a marker that the surrounding function is a startup or shutdown hook rather than a request-handling endpoint; calls of the form `lc.Append(fx.Hook{OnStart: ..., OnStop: ...})` SHALL NOT themselves emit an Abstract_Input or Abstract_Output.

### Requirement 13: Treat viper Configuration Reads as Configuration, Not Inputs

**User Story:** As an MCP_Client, I want the Project_Analyzer to ignore `viper` configuration reads when extracting Abstract_Inputs, so that environment variables and configuration file keys do not pollute the Project_Profile's input list with values that govern startup parameterization rather than runtime data.

#### Acceptance Criteria

1. WHEN the Project_Analyzer encounters a call to any function or method exported by the Go_Config_Library (`github.com/spf13/viper`), including but not limited to `viper.New()`, `viper.SetConfigName(<name>)`, `viper.SetConfigType(<type>)`, `viper.AddConfigPath(<path>)`, `viper.ReadInConfig()`, `viper.AutomaticEnv()`, `viper.SetEnvPrefix(<prefix>)`, `viper.BindEnv(<key>, ...)`, `viper.WatchConfig()`, `viper.GetString(<key>)`, `viper.GetInt(<key>)`, `viper.GetBool(<key>)`, `viper.GetDuration(<key>)`, `viper.UnmarshalKey(...)`, and the equivalent methods on a `*viper.Viper` instance produced by `viper.New()`, THE Project_Analyzer SHALL treat the call as a configuration read and SHALL NOT emit an Abstract_Input from that call.
2. THE Project_Analyzer SHALL NOT use any string literal passed to a Go_Config_Library call as a route path, schedule expression, message destination, file path, or table name candidate for Requirements 3 through 9. The string literal `"app"` passed to `viper.SetConfigName("app")` is a configuration filename, not a route or table.
3. WHERE a Go_Source_File reads a configuration value from a Go_Config_Library call and passes that value to a construction recognized by Requirements 3 through 9 (for example, `cfg.JobCfg.CronSchedule` passed to `c.AddFunc(...)` in `cat-service`, or `cfg.JMSConfig.PaymentErrQueue.Destination` passed to `domain.Message{Destination: ...}` in `repayment_service`), THE Project_Analyzer SHALL still emit the Abstract_Input or Abstract_Output for the construction itself, recording the dynamic value as `<dynamic>` per the relevant requirement, because the construction is the runtime entry point even when its parameters are configuration-driven.

## Open Questions

The following constructions are supported by the Go language or by libraries listed in the user's stack but do not currently appear in any of the four sample repositories. They are recorded here so that the design phase can decide whether to include them in the initial implementation or defer to a follow-up.

1. **Go 1.22+ method-prefixed `http.ServeMux` patterns.** Requirement 3.3 specifies how to parse them, but no sample currently uses the `"GET /users/{id}"` form. All observed registrations are bare paths (`"/healthz"`, `"/debug/pprof/"`, `"/"`). The implementation should still handle the method-prefixed form, since the user's stack pins Go 1.24 and existing repositories may adopt the new syntax in future commits.
2. **Direct `stomp.Dial(...)` or `stomp.Connect(...)` calls.** All ActiveMQ access in the four sample repositories goes through the in-house `esb-go-libs/activemq` wrapper; no repository imports `github.com/go-stomp/stomp/v3` directly. The design phase may consider adding a fallback recognizer for direct STOMP calls, but it is not required for the four sample repositories.
3. **gRPC service registrations.** The sample repositories `fec_pool_service` and `repayment_service` register gRPC services with `pb.RegisterPoolAPIServer(grpcsrv, ...)`, `pb.RegisterRepaymentServiceAPIServer(grpcsrv, ...)`, and similar calls, and mount gRPC-gateway HTTP bridges with `pb.RegisterPoolAPIHandlerFromEndpoint(...)`. These calls expose RPC endpoints that are conceptually Abstract_Inputs but are not covered by any requirement above. Whether to treat gRPC server registrations as a new requirement (analogous to Requirement 3 for HTTP) is deferred to the design phase.
4. **Direct Oracle access via `github.com/sijms/go-ora/v2`.** The `fec_pool_service` repository itself uses this driver to talk to Oracle, but no other repository does. Recognizing direct Oracle SQL execution would require duplicating the SQL-extraction logic of Requirement 9 against `database/sql`-style call shapes; this is deferred to the design phase since only one repository in the user's stack would benefit.
5. **Kafka and Redis usage.** Both `github.com/segmentio/kafka-go` and `github.com/redis/go-redis/v9` appear as indirect dependencies in several `go.mod` files (consumed transitively through `esb-go-libs`). No application repository calls these libraries directly in its first-party source. They are intentionally not detected.
6. **Representing the unknown database access mode in the parent spec.** Requirement 9.8 directs the Project_Analyzer to still emit a Database_Table_Dependency when a table name is identifiable but the SQL keyword cannot be matched to any of `SELECT`, `JOIN`, `INSERT`, `UPDATE`, `DELETE`, `MERGE`, or `CREATE TABLE`. The parent spec's `DatabaseAccessMode` enum is currently closed at `read | write | read_write` (defined in `src/project_knowledge_mcp/models.py`) and rejects any other value. The design phase must resolve how to represent the unknown-access-mode case — adding `unknown` to the parent spec's enum, choosing a conservative default such as `read_write`, or suppressing the single dependency — and must update the parent spec `project-knowledge-mcp` accordingly. This is a follow-up requirement on the parent spec, not a runtime extension introduced by this feature.
